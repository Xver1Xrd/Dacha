# СНТ «Михайловское» — сайт + бэкенд

Лендинг услуг по работе на участках с интерактивной картой работ и админпанелью.
Фронтенд — статика (HTML/CSS/JS), бэкенд — на Go: стандартная библиотека плюс
`pgx` для режима PostgreSQL (файловый режим `data.json` внешних зависимостей не требует).

## Запуск

```bash
go run .
# или собрать бинарник:
go build -o dacha . && ./dacha
```

Сервер поднимется на http://localhost:8080 (порт можно сменить: `PORT=3000 go run .`).
Он раздаёт сайт и REST API. Открывайте сайт по адресу сервера, не через `file://`.

При первом запуске рядом создаётся `data.json` со стартовыми данными.
Стартовые аллеи/работники/отзывы и логин/пароль администратора — в `seed.go`,
общие для файлового хранилища (`main.go`) и PostgreSQL (`store.go`), чтобы не
дублироваться в двух местах.

## Что где

- `index.html` — сайт (публичной карты пока нет — она не доработана и доступна только в админке).
- `admin.html` — админпанель (вход, редактор карты, работники, администраторы, отзывы).
- `css/`, `js/` — стили и фронтенд-логика. `js/store.js` — клиент API.
- `main.go` — сервер: статика + API. `store.go` — хранилище PostgreSQL. `seed.go` — общие стартовые данные.
- `sessions.go`, `ratelimit.go`, `middleware.go`, `validate.go` — сессии, лимитер входа, security-заголовки/CSRF, лимиты ввода.
- `data.json` — файловое хранилище для локальной разработки (создаётся автоматически, в git не коммитится).

## API (кратко)

| Метод | Путь | Доступ | Назначение |
|------|------|--------|------------|
| GET | `/api/data` | все | данные карты (аллеи, работы, работники) |
| GET | `/api/reviews` | все | список отзывов |
| POST | `/api/login` | все | вход (ставит cookie-сессию и csrf-cookie) |
| POST | `/api/logout` | все | выход |
| GET | `/api/session` | все | проверка текущей сессии |
| GET/POST | `/api/admins` · DELETE `/api/admins/{login}` | админ | управление администраторами |
| POST `/api/workers` · PUT/DELETE `/api/workers/{id}` | админ | управление работниками |
| POST `/api/reviews` · PUT/DELETE `/api/reviews/{id}` | админ | управление отзывами |
| POST | `/api/works` | админ | статус/инфо участка (`rec:null` — сброс) |

Все мутирующие запросы (POST/PUT/DELETE) от авторизованного администратора должны
дублировать значение cookie `snt_csrf` в заголовке `X-CSRF-Token` — так `js/store.js`
делает автоматически.

## Безопасность

- **Пароли** — PBKDF2-HMAC-SHA256 (соль + 100 000 итераций), не в открытом виде.
  Пароль администратора — минимум 6 символов.
- **Сессии** — случайный токен в HttpOnly-cookie, время жизни 24 часа (`sessionTTL`
  в `sessions.go`). В режиме PostgreSQL сессии хранятся в таблице `sessions` и
  переживают перезапуск/передеплой процесса; в файловом режиме (локальная
  разработка) — в памяти, поэтому перезапуск разлогинивает администраторов.
- **Secure-cookie включается автоматически** — сервер сам определяет HTTPS
  (по `r.TLS` или заголовку `X-Forwarded-Proto`, который выставляют обычные
  обратные прокси/хостинги) и не требует ручной правки кода.
- **CSRF** — защита схемой double-submit cookie (`snt_csrf` + заголовок
  `X-CSRF-Token`) для всех мутирующих запросов авторизованных администраторов.
- **Rate limiting** — не более 5 неудачных попыток входа за 15 минут на пару
  IP+логин, дальше блокировка на 15 минут (`ratelimit.go`).
- **Заголовки безопасности** — CSP, `X-Content-Type-Options`, `X-Frame-Options`,
  `Referrer-Policy`, `Permissions-Policy` (`middleware.go`).
- **Валидация ввода** — ограничения длины на логин/пароль/имя/телефон/текст
  отзыва и т.п. (`validate.go`), тело запроса ограничено 1 МиБ.
- Для HTTPS в продакшене всё равно нужен обратный прокси (nginx/Caddy) или
  хостинг, который сам терминирует TLS (Render/Railway/Fly.io) — сервер сам
  TLS не поднимает.

## Тесты и CI

```bash
go test ./...
go vet ./...
```

Тесты покрывают хеширование паролей, файловое хранилище (CRUD работников/
отзывов/администраторов), лимитер попыток входа, сессии и полный HTTP-цикл
логин → CSRF → мутирующий запрос (`*_test.go`). GitHub Actions
(`.github/workflows/ci.yml`) на каждый push/PR гоняет `go build`, `go vet`,
`gofmt -l` и `go test -race`.

## Хостинг

GitHub Pages не подходит (там только статика). Нужен хост, где можно запустить бинарник:
Render / Railway / Fly.io / собственный VPS. Соберите `go build` и запустите бинарник,
положив рядом записываемый `data.json` (для файлового режима).

### Railway

В репозитории есть `railway.json` (сборка через Nixpacks: `go build -o dacha .`,
запуск `./dacha`, healthcheck на `/api/data`). Достаточно:

1. Подключить репозиторий как сервис в Railway.
2. Добавить плагин **PostgreSQL** — Railway сам положит `DATABASE_URL` в
   переменные сервиса, сервер переключится в режим Postgres автоматически.
3. Без Postgres сервер поднимется в файловом режиме, но `data.json` живёт на
   эфемерном диске — при каждом передеплое данные и сессии сбросятся.

`PORT` и HTTPS (через `X-Forwarded-Proto`) Railway настраивает сама, ничего
дополнительно указывать не нужно.
